## Summary
### GitHub Repository Migration Update:
The team discussed the successful migration of 22 repositories to the new GitHub organization github.com/openmoonray, with Jon leading the migration effort. Jean-Francois advised disabling dangerous pull request targeting at the organization level and suggested creating a LF help desk ticket for guidance on repository settings. The team also discussed implementing Easy CLA for forks, with Matthew confirming this could be managed independently once the organization was set up, and Jon noted that John Mertek was helping with website and documentation updates to replace DreamWorks content with face-to-face materials. Finally, Jon outlined plans to migrate internal JIRA tickets to the new GitHub issues system and implement better labeling practices.

### JIRA Transition and Versioning Strategy:
The team discussed transitioning away from JIRA integration, with Jean-Francois confirming that the ASWF JIRA instance may have been decommissioned. Jon suggested recreating public-facing issues while keeping internal bug reports separate, and the group agreed this approach made sense given the need to groom their issues database. The discussion then shifted to versioning and release strategies, where Jon and Randy Packer raised concerns about their current four-part versioning scheme (major, minor, patch, and build) tied to production schedules, suggesting this needs reconsideration for open source compatibility.

### Versioning and CI/CD Planning:
Jon and Matthew discussed versioning schemes for their project, considering the current practice of weekly internal releases versus potential changes for external users. They explored aligning major version bumps with show locks or reference platform updates, but decided to maintain the current approach for now as there is no urgent need to change. The conversation then shifted to CI/CD needs, with Jon outlining the requirement to support multiple reference platforms and suites, and noting that establishing basic build and unit tests, potentially starting with Conan recipes for dependencies, would be a good first step.

### 2023-2027 Image Update Planning:
Jean-Francois discussed updating the 2023 images and working on a 2027 set, aiming to have updated images ready within a week. He suggested Jon start by building in the CI-VFX-ALL image to identify missing dependencies and open an issue in the ASWF Docker repo to help create a specific build image for OpenMoonRay. Jean-Francois clarified that the Conan images are currently Linux-only and explained the differences between building in a controlled VFX platform environment versus using GitHub's latest runners. Jon mentioned an existing Mac build recipe using CMake's external project add to download and build dependencies, which could be a potential alternative approach.

### GitHub Action Runners Configuration:
Jean-Francois explained that GitHub action runners are ephemeral VMs with no persistent storage unless explicitly configured with caching. He reviewed the dependencies needed for Open Moonray, noting that most are already available except for a few specific libraries like JSON-CPP, ISPC, Embry, and open image denoise. For non-Linux builds, Jean-Francois outlined plans to build Conan packages under Windows and macOS, with a medium-term goal to provide pre-built Conan packages. Jon agreed to investigate OpenImageIO's setup and reach out to Larry for guidance, while Paul indicated he would discuss potential involvement in the project after his meeting with Mark.

### Hydra 1 to 2 Transition Plan:
Rob discussed the successful implementation of the Hydra 2 to Hydra 1 shim and proposed dropping support for Hydra 1, with a potential cutoff at version 25.5. He sought feedback on supporting Moonray-specific features through Hydra 2 and considered publishing proposals for external review. Matthew suggested sharing the proposals with the USD Working Group on Slack for additional feedback. Duncan provided an update on their progress with Hydra 2 in version 25.8, noting they are moving away from Hydra 1 workflows to support external renderers like Moonray.

### Hydra 2 Support and Integration:
The team discussed Hydra 2 support and schema integration, with Duncan noting that Katana requires schemas to populate their UI properly. Rob proposed writing notes on the topic to generate feedback from various teams. The group also addressed Windows build plans, with Alex confirming that Windows code is coming soon with a new release of Moonracer. Jon and Jean-Francois discussed CI setup options, including the use of Ubuntu runners and potential limitations of free runners for building. They explored alternatives like using pay runners or managing custom OS images, with Jean-Francois mentioning the FVDB project as an example of managing custom GPU instances.

### Container Package Building Demonstration:
Jean-Francois demonstrated the use of containers for building packages, explaining how they provide a stable environment compared to GitHub's raw runners which can change configurations without notice. He showed examples of container management, including the decision of whether to use year-based or specific release-based versions, and discussed different classes of Conan packages including wrappers, vendored packages, and custom recipes. The discussion concluded with Jean-Francois noting that while building OpenMoonRay in the Conan environment would be interesting, the immediate priority is to build the required packages first.

### Conan and CMake Integration Discussion:
Jean-Francois and Jon discussed the use of Conan packages and CMake presets in building projects like Open Moonray. Jean-Francois explained how Conan generates CMake harnesses to help consumers find packages correctly, but noted challenges with custom find modules and include paths not matching standard CMake variables. Jon shared how Open Moonray uses CMake presets and environment variables to handle dependencies flexibly, and they both acknowledged the potential for modernizing CMake usage. They also briefly discussed Jon's efforts to reduce dependencies, particularly on Boost, across multiple repositories.

### ASWF Open Moonray Project Discussion:
Jean-Francois and Jon discussed the ASWF Docker repo and Open Moonray project. Jean-Francois provided background on the project's history and his involvement with the Visual Effects Society Tech Committee. Jon shared his experience managing the open source side of Moonray, including the build system and integration of contributions. They discussed challenges with security vulnerabilities in OpenEXR and GitHub's handling of increased traffic. Jean-Francois suggested setting up SonarCloud for static analysis and offered to help Jon get started with the process. Jon expressed interest in reaching out to Jean-Francois for further guidance as he continues working on implementing CI for Open Moonray.
